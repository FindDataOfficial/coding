# Schema Creator

Generate Python SQLAlchemy model files from class diagrams in `diagram/`.
The output is production-ready ORM code — a direct translation from design
to implementation.

## Where things live

- **Input**: `diagram/` — class diagrams in YAML format (generated by diagram-creator)
- **Context**: `goal/initial-goal/` — original requirements and success criteria (primary context)
- **Context**: `goal/clear-goal/` — design decisions from brainstorming (extra context, may not exist)
- **Output**: `schema/` — Python files with SQLAlchemy models
- **Directory structure**: Mirror the input (case-insensitive). `diagram/paas/mcp-diagram.yaml` → `schema/paas/mcp.py`

## Before you start

1. Read the diagram YAML file to understand the class structure
2. **Read the corresponding initial-goal file** — The initial-goal has the full requirements. Derive the path by mirroring: `diagram/<path>/<name>-diagram.yaml` → `goal/initial-goal/<path>/<name>.md` (case-insensitive).
3. **Read the corresponding clear-goal file** — The clear-goal contains refined design decisions (e.g., "registry uses YAML not SQLite"). Derive: `diagram/<path>/<name>-diagram.yaml` → `goal/clear-goal/<path>/<name>.md` (case-insensitive). If it doesn't exist, proceed without it.
4. Identify which classes are **data entities** (persistable) vs **service classes** (runtime-only), using both the diagram attributes AND the goal context
5. Check `schema/` for any existing schema files that relate to the current diagram

## How to tell data entities from service classes

Not every class in a diagram becomes a database table. Use this heuristic:

- **Data entity** — attributes are scalars (String, int, boolean, float),
  foreign keys, or lists of other data entities. Methods are simple CRUD or
  serialization (toDict, fromDict). These become SQLAlchemy models.
- **Service class** — attributes are references to other services/managers,
  config paths, or runtime state (e.g., `runningProcesses`, `checkInterval`,
  `configDir`, `templatePath`, `dockerAvailable`). Methods do orchestration
  (start, stop, deploy, reload, allocate). These are NOT models — they're
  application code, not schema.

Only generate SQLAlchemy models for data entities. For service classes, add
a comment at the bottom of the output noting they were skipped and why.

**Single-model schemas are valid.** If only one class qualifies as a data
entity, that's fine — don't force extra models. Many systems have one
persistence table and many runtime services around it.

## Type mapping

Map diagram types to SQLAlchemy column types:

| Diagram type | SQLAlchemy type | Notes |
|---|---|---|
| `String` | `sqlalchemy.String` | Default length 255 unless it's an ID or code (then 64) or a path (then 512) |
| `int` | `sqlalchemy.Integer` | |
| `boolean` | `sqlalchemy.Boolean` | |
| `float` | `sqlalchemy.Float` | |
| `Map<String, X>` | `sqlalchemy.JSON` | Dict/map types become JSON columns |
| `List<Entity>` | `relationship()` | See relationships section below |
| Custom types (e.g., `RuntimeConfig`) | `sqlalchemy.JSON` | If the type is a value object (not in the diagram as its own class), store as JSON. Add a comment explaining the choice. |
| External types (e.g., `Process`) | skip | Types from external libraries (subprocess, os, etc.) are runtime-only. Don't map — the class is a service, not an entity. |

### String length conventions

Pick the right length based on what the column stores:

- **IDs / codes**: `String(64)` — primary keys, foreign keys, short identifiers
- **Names / labels**: `String(255)` — human-readable names
- **Paths / URLs**: `String(512)` — filesystem paths, health URLs, endpoints
- **Everything else**: `String(255)` — the default

## Relationship mapping

Read the `relationships` section of the diagram to determine how entities
connect:

- **composition** (strong ownership): Child gets `parent_id` ForeignKey with
  `ondelete="CASCADE"`. Parent gets a `relationship()` with
  `cascade="all, delete-orphan"`.
- **association** (weak reference): "From" side gets a ForeignKey to "to".
  Both sides get a `relationship()` with `backref`.
- **inheritance**: Use SQLAlchemy joined table inheritance — child model
  inherits from parent, both have `__tablename__`.
- **dependency**: Not mapped to the database. Add a comment noting the
  dependency exists in code but not in schema.

Only map relationships where **both endpoints are data entities**. If one
side is a service class, note the relationship as a comment in the skipped
section but don't create a ForeignKey.

## Output format

Generate a standard Python file with these sections:

```python
"""
SQLAlchemy schema for <system-name>
Generated from <diagram-path>
"""
from sqlalchemy import Column, String, Integer, Boolean, Float, JSON, ForeignKey
from sqlalchemy.orm import relationship, declarative_base

Base = declarative_base()


# ============================================
# Data entities
# ============================================

class EntityName(Base):
    __tablename__ = "entity_name"

    id = Column(String(64), primary_key=True)
    name = Column(String(255), nullable=False)
    # ... other columns ...
    # ... relationships ...


# ============================================
# Skipped service classes (not persistable)
# ============================================
# ClassName: reason skipped

# ============================================
# Skipped relationships (not both data entities)
# ============================================
# SourceClass --type--> TargetClass: reason not mapped
```

- **Columns use snake_case** (e.g., `sourcePath` → `source_path`)
- **Non-nullable columns** get `nullable=False` when the attribute is
  clearly required (name, path, port). Nullable columns (runtime state like
  `process_id`, optional config like `runtime_config`) get `nullable=True`.
- **Default values** use `default=` for known defaults (e.g.,
  `default="stopped"` for status, `default="server:app"` for entry_point).
  Don't guess defaults — only set them when the diagram or clear-goal
  implies a sensible default.

## Naming conventions

- **Table names**: snake_case of the class name (e.g., `MCPServer` →
  `mcp_server`, `PortAllocator` → `port_allocator`)
- **Column names**: snake_case of the attribute name (e.g., `sourcePath` →
  `source_path`, `healthUrl` → `health_url`)
- **File names**: match the source diagram without `-diagram` (e.g.,
  `mcp-diagram.yaml` → `mcp.py`, `order-system.yaml` → `order_system.py`)
- **Primary key**: use `id` as the PK column name, typed as `String(64)`

## Workflow

### Phase 1: Analyze

1. Read the diagram YAML file
2. Derive and read the corresponding clear-goal file (mirror path,
   case-insensitive: `diagram/<path>/<name>-diagram.yaml` →
   `goal/clear-goal/<path>/<name>.md`). If not found at the derived path,
   search `goal/clear-goal/` for a matching filename.
3. For each class, classify it as data entity or service class using the
   heuristic above AND the clear-goal context (design decisions may
   override the heuristic).
4. For data entities, list all attributes and their mapped SQLAlchemy types.
5. For relationships, determine which ones involve two data entities (these
   get ForeignKeys) vs one entity + one service (comment only).

### Phase 2: Confirm with the user

Present a summary before generating code:

- Which classes become models and why
- Which classes are skipped and why
- The relationship mapping decisions
- The expected table count (it's OK if it's 1)

This is the moment to catch mistakes — the user might want a class to be a
model even though it looks like a service, or vice versa. Don't skip this
step.

### Phase 3: Generate

Once confirmed, write the Python file to `schema/<mirror-path>.py`.

**Quality checks before saving:**
- Every data entity has `__tablename__` in snake_case
- Every attribute has a mapped SQLAlchemy type (no unmapped custom types
  without explanation)
- Every composition/association relationship between two data entities has
  the correct ForeignKey + relationship on both sides
- Table names follow the snake_case convention
- The file imports all needed SQLAlchemy types (no missing imports, no
  unused imports)
- `Base = declarative_base()` is defined exactly once at the top
- Non-nullable columns have `nullable=False`

### Phase 4: Validate

1. **Completeness**: Every data entity class from the diagram is present as
   a model
2. **Type correctness**: No unmapped types (every attribute has a concrete
   SQLAlchemy column type or a comment explaining why it was skipped)
3. **Relationship integrity**: Every ForeignKey references an existing table
   name
4. **Python syntax**: The file parses without errors
   (`python -c "import ast; ast.parse(open('...').read())"`)

### Phase 5: Offer to continue the pipeline

After saving the schema, **ask the user**: "Schema saved. Want to generate the service implementation code now? I can run `/code-creator` with this diagram and schema."

If the user says yes, invoke the Skill tool with `skill: "code-creator"` and pass the diagram file path as args. This chains the pipeline: schema → code.

If the user says no or "later", that's fine — they can always run code-creator separately.
